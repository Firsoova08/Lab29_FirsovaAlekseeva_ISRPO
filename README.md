## Краткое описание работы

В ходе данной лабораторной работы был разработан RESTful API для управления списком задач (ToDo List) с использованием контроллеров в ASP.NET Core. Целью работы было изучение архитектурного стиля REST, принципов создания CRUD-операций, различий между Minimal API и Controller-based API, а также освоение инструментов тестирования (Swagger UI, REST Client) и правильного использования HTTP-статусов.

## Список реализованных маршрутов

| Метод   | Маршрут                                | Действие                                    |
|---------|----------------------------------------|---------------------------------------------|
| GET     | /api/tasks                             | Получить все задачи (с фильтром по `completed`) |
| GET     | /api/tasks/{id}                        | Получить задачу по ID                       |

| DELETE  | /api/tasks/{id}                        | Удалить задачу                              |

## Итоговая таблица: ASP.NET Core Controller-based API

| Аспект                 | ASP.NET Core Controllers                                                              |
|------------------------|---------------------------------------------------------------------------------------|
| **Маршруты**           | Атрибут `[HttpGet]`, `[HttpPost]` и т.д. над методом                                  |
| **Группировка маршрутов** | Класс-контроллер, наследующий `ControllerBase`                                    |
| **Базовый URL**        | `[Route("api/[controller]")]` (например, `api/tasks`)                                  |
| **Параметр пути**      | `int id` — параметр метода (например, `GetById(int id)`)                              |
| **Параметр запроса**   | `[FromQuery] bool? completed`                                                         |
| **Тело запроса**       | `[FromBody] CreateTaskDto dto`                                                        |
| **Ответ 200**          | `return Ok(data)`                                                                     |
| **Ответ 201**          | `return CreatedAtAction(nameof(GetById), new { id = task.Id }, task)`                 |
| **Ответ 404**          | `return NotFound(new { Message = "..." })`                                            |
| **Ответ 204**          | `return NoContent()`                                                                  |
| **Типизация данных**   | Строгая (C#) — компилятор проверяет типы на этапе написания кода                      |
| **Документация**       | Swagger / OpenAPI (устанавливается отдельно через `Swashbuckle.AspNetCore`)          |

## Главные выводы

1.  **REST — это архитектурный стиль, а не протокол.** Его принципы (stateless, единый интерфейс, кэшируемость) универсальны и применимы к API на любом языке программирования. Главное правило RESTful API: ресурс идентифицируется URL, а действие над ним — HTTP-методом.

2.  **Controller-based API обеспечивает лучшую организацию кода по сравнению с Minimal API.** В крупных проектах разбиение логики по отдельным классам-контроллерам (например, `TasksController`, `UsersController`) делает код более поддерживаемым и масштабируемым, в отличие от хранения всех маршрутов в `Program.cs`.

3.  **Использование DTO (Data Transfer Object) критически важно для безопасности и контроля данных.** DTO позволяют серверу принимать от клиента только те поля, которые разрешены (например, `Title`, `Description`), игнорируя служебные поля (`Id`, `CreatedAt`), которые генерируются автоматически. Это защищает API от некорректных или вредоносных входных данных.

4.  **Правильные HTTP-статусы являются неотъемлемой частью контракта API.** Клиент (например, JavaScript на фронтенде) должен иметь возможность однозначно определить результат операции по коду ответа: `201 Created` — ресурс создан, `400 Bad Request` — ошибка в данных клиента, `404 Not Found` — ресурс не существует. Возврат всегда `200 OK` лишает клиента этой важной информации.
